# Project Context — shorts (autonomous crew workflow, M1)

## Pitch
An overseer-driven crew that pushes `pipeline_v2.py` forward — landing the
hook scorer, selective subtitles, an automated eval/QC loop, and a
multi-variant A/B emitter — without the user babysitting it.

## Audience & done-criteria
- **Audience:** jperr only. Renders are the user's own gameplay shorts. No
  external customers in M1; no uploads to TikTok/YT/IG.
- **M1 done:** hook scorer wired into v2 ranking, selective-subtitle
  prototype rendered against the eval suite, automated eval loop scoring
  every render, multi-variant A/B emitter producing 2 variants per moment,
  all merged to `main` with watchable artifacts in `delivered/`.

## Stack
- **Language:** python3, single-file pipeline_v2.py style. No packaging
  refactor in M1.
- **Inference budget:** local-first per `shorts-plan.md:88-101`. The two
  semantic-judgment lanes (engaging-span detection, hook-overlay text)
  run as **dedicated crew members** with their own sessions, not as
  inline Haiku API calls. Sonnet only on demonstrated failure.
- **Render contract:** results land in
  `/Users/jperr/Documents/shorts/delivered/<bead-id>-<timestamp>-<stem>.mp4`
  via `pipeline_v2.deliver()`. Bad renders are moved to
  `delivered/rejected/<reason>/` by the eval loop. Nothing in `delivered/`
  is ever overwritten or deleted by the crew.

## Scope (milestone 1)

### In
- Wire v1 hook scorer into v2 ranker (port `HOOK_WORDS`, `hook_score`,
  rank() integration from `pipeline.py`).
- Variety / min-gap re-ranker + `hook_in_first_3s`, `standalone_3s`,
  `duration_fit` features on top of the merged scorer.
- Crew-driven "engaging span" detector (D-04 path A: local signals first;
  delegated semantic judgment on hot candidates).
- Selective subtitles — burned only on engaging spans, blank elsewhere.
  Word-level karaoke as the rendering primitive underneath.

- Auto eval/QC loop: per-render metrics (loudness, face-tile occupancy,
  reframe jerk, hook-window energy, transcript coverage on hot spans),

## Decisions (locked)

- **D-01 Autonomy ceiling:** A — crew merges code and lands renders in
  `delivered/` autonomously. User reviews after the fact.
  *Reason:* overseer is an explicit long-running mode; renders are cheap
  to delete; the delivery contract preserves a watchable artifact for
  post-hoc review.

- **D-02 Rollback policy:** B — eval loop moves failed renders to
  `delivered/rejected/<reason>/`. Main `delivered/` stays clean. Crew
  never deletes from `delivered/`.

- **D-03 "Bad render" definition:**
  - Hard fails (auto-reject to `rejected/`): file <100KB, duration <25s
    or >65s, loudnorm I outside [-15,-13], face-tile black >50% of
    frames, transcript empty.
  - Soft flags (warn, keep in `delivered/`): reframe jerk above
    threshold, hook-window energy below threshold, no interjection in
    first 3s.

- **D-04 Selective-subtitles definition:** A (local signals) plus a
  dedicated **crew lane** that makes the semantic engagement call on
  candidate spans. Lane spec is forge's call. Selective-burn writes
  subtitle ASS only for spans tagged `engaging=true`; elsewhere the
  subtitle track is empty.
  *Reason:* gullivan confirms no commercial product does selective burn
  — this is the differentiator. Keeping the judgment in a dedicated crew
  session (not inline Haiku) keeps the loop debug-able and the prompt
  iteration visible.

- **D-05 Hook overlay generation:** B-equivalent — one overlay text per
  shortlisted clip, generated by a dedicated **crew lane** (not inline
  Haiku). Rendered as ASS title.

- **D-06 Multi-variant A/B emitter:** A — 2 variants per moment, same
  source clip, trim windows differ by lead +/- 5s. Overlay-variant (B)
  becomes M1.5 once D-05 lands; trim×overlay grid (C) deferred.

- **D-07 Branch / worktree strategy:** **OVERRIDE — no polecat
  worktrees, no per-bead branches.** Everything commits straight to
  `main`. `delivered/` stays where it is. README.md and shorts-plan.md
  are updated by T-02 to remove polecat semantics.

- **D-08 Eval VOD suite:** the three VODs above, fetched
  deterministically by `scripts/fetch_sources.sh` (or Makefile target).
  Crew never substitutes a different VOD without a new bead.

- **D-09 Crew lanes:** **forge proposes the lineup.** Engagement
  judgment (D-04) and overlay text (D-05) are confirmed needs; render
  execution, scorer porting, eval/grader, subtitle engineering are
  candidate lanes — final shape is forge's call. T-01 is the forge ask.

- **D-10 Brutus gating:** on. Every code-shipping bead routes through
  brutus before merge. Brutus runs the smoke clip on the eval suite and
  verifies the `delivered/` contract holds.

## Coordination tradeoff (open — flag, don't block)

`commit-everything-to-main` (D-07) plus parallel crew lanes creates a
merge-conflict surface. We have **not** defined a branch policy because
the user explicitly said no branches. Candidate serialization
mechanisms, in order of preference:

1. **One-bead-in-flight rule per file region.** Beads carry an
   approximate `surface` label (e.g. `ranker`, `compose`, `subtitles`,
   `eval`). Athena gates so two in-progress beads never share a
   surface. Cheap and matches how the file already partitions.
2. **Brutus serializes commits.** Crew members open a bead, do work in
   a scratch dir, hand the diff to brutus; brutus rebases on main,
   runs smoke, commits. Single writer to main.
3. **`git pull --rebase` before every commit, conflict-fail loud.**
   Last-resort fallback if (1)/(2) prove too restrictive.

Recommendation: start with (1). Forge should bake the surface-label
convention into the lane briefs. Revisit if conflicts appear.
